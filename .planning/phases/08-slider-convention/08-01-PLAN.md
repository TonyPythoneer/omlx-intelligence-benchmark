<objective>
Fix the visibly broken PARAMS range slider in `FilterBar.vue` by replacing the two
overlapping native `<input type="range">` elements with a reusable, reka-ui-backed
`ui/slider.vue` (dual-thumb range), and establish "reka-ui headless primitives for
interactive widgets" as the documented project convention.

Purpose: The PARAMS slider renders as two disconnected empty circles (no track, no
fill) on WebKit — the project's first and worst symptom of `reka-ui` being a declared
but entirely unused dependency. This phase fixes the bug with the correct headless
primitive and sets the precedent that Phase 09 (dialog/select) follows.

Output:
- NEW `src/components/ui/slider.vue` (reka-ui `SliderRoot`/`SliderTrack`/`SliderRange`/`SliderThumb`, cva+cn styled)
- EDITED `src/components/FilterBar.vue` (PARAMS block uses `<UiSlider>`; public props/emits unchanged)
- EDITED `CLAUDE.md` (convention documented)

Scope guardrails (do NOT do here): no dialog/select migration (Phase 09), no leaf-component
rewrites, no data/feature/JSON-contract change, no new tooling, no `App*` shim, no manual
variant maps. `reka-ui@^2.9.9` is already in `package.json` — this phase installs NOTHING.
</objective>

<key_facts>
- `reka-ui@2.9.9` exports `SliderRoot`, `SliderTrack`, `SliderRange`, `SliderThumb` (all verified present). v-model / `modelValue` is `number[]`; array length = thumb count; two-thumb range = `[min, max]`. Props: `min`, `max`, `step`, `minStepsBetweenThumbs`, `orientation`, `dir`, `disabled`, `inverted`. `SliderRange` is the filled segment; `SliderTrack` is the rail; one `SliderThumb` per model entry. Keyboard a11y (arrows, Home/End) and aria come for free.
- `cn()` lives at `@/lib/utils` (clsx + tailwind-merge). The canonical styling contract in this repo is `cva` + `VariantProps` + `cn()` (see `button.vue`). NO `App*` shim, NO manual variant maps.
- ui/ components are imported PER-FILE (e.g. `import UiCard from './ui/card.vue'`). There is NO `ui/index.*` barrel and no global registration — so "registration" for the new slider is just adding a local import in `FilterBar.vue`.
- The intended thumb design (port from the broken markup): `w-4 h-4 rounded-full bg-primary border-2 border-white shadow`. The intended rail: `bg-border rounded-full h-1` (use ~`h-1.5` for visibility); the fill: `bg-primary`.
- FilterBar's preserved contract: props `paramsMinIdx:number`, `paramsMaxIdx:number`, `parametersBreakpoints:number[]`; emits `update:paramsMinIdx`, `update:paramsMaxIdx`. Local mirrors `paramsMinIdxLocal`/`paramsMaxIdxLocal`, two `immediate` prop→local watchers, and the `[min,max]` swap-guard watcher (emits swapped values when min>max). `paramsLabelAt(idx)` returns `Inf` when `idx >= breakpoints.length`, else `${breakpoints[idx]}B`. Slider domain is index-based: `0 … parametersBreakpoints.length` (inclusive upper = "Inf").
- No Vitest test imports `FilterBar.vue` or the slider (tests cover only `import.mjs`, `useImport`, `useLabeling`). `vp test` is therefore a pure no-regression gate here, not a behaviour check for the slider.
</key_facts>
