<task type="auto">
  <name>Task 1: Create useLabeling composable with state management and validation</name>
  <files>src/composables/useLabeling.ts</files>
  <action>
Create a new composable file `src/composables/useLabeling.ts` that provides:

1. **State refs:**
   - `isLabelingMode: Ref<boolean>` — true when user has clicked Label button
   - `isDirty: Ref<boolean>` — true when mutableEntries have been modified (from Import Apply OR label edits)
   - `labelEdits: Ref<Record<string, Partial<Entry>>>` — per-model accumulated edits, shape is Entry fields that can be edited (spec, abilities, tiers, deprecated)

2. **Validation logic** (pure function, exported for testing):
   - `validateEditField(field: string, value: any): string[]` — returns array of error messages (empty if valid)
     - `parameters_b`: must be null or a non-negative number
     - `size_gb`: must be null or a non-negative number
     - `quantization`: any non-empty string is valid (or null)
     - `abilities.thinking`: any boolean is valid
     - `abilities.mtp`: any boolean is valid
     - `deprecated`: any boolean is valid
     - `tiers.*`: any boolean is valid

3. **Computed refs:**
   - `validationErrors: ComputedRef<Record<string, Record<string, string[]>>>` — per-model, per-field validation errors
   - `hasValidationErrors: ComputedRef<boolean>` — true if any field has errors

4. **Functions:**
   - `setDirty()` — explicitly mark isDirty = true (called when Import Apply merges entries)
   - `updateLabelEdit(modelName: string, field: string, value: any)` — update labelEdits[modelName][field], trigger validation
   - `commitLabelEdits(mutableEntries: Ref<Entry[]>)` — apply all labelEdits to mutableEntries, reset labelEdits, clear isDirty, exit labeling mode
   - `cancelLabelEdits()` — discard labelEdits, exit labeling mode, keep isDirty true (user may save export anyway)
   - `toggleLabelingMode()` — flip isLabelingMode

**Validation field structure:** Store validation errors per model and per field so BenchmarkTable can show inline error messages next to the problematic field.

**Notes:**
- Keep labelEdits immutable: update via computed assignments, not mutations
- Export validateEditField for unit testing
- Do not export state refs directly; export via destructuring in return object (per Vue 3 pattern)
  </action>

     - `<input type="text" v-model="labelEdits[model].spec.quantization" @change="updateEdit(model, 'spec.quantization', ...)">`
     - `<input type="number" v-model.number="labelEdits[model].spec.size_gb" @change="updateEdit(model, 'spec.size_gb', ...)">`

   - Tiers checkboxes:
     - `<input type="checkbox" v-model="labelEdits[model].tiers.opus" @change="...">`
     - `<input type="checkbox" v-model="labelEdits[model].tiers.sonnet" @change="...">`

         </div>
       </td>
       <!-- ... more fields ... -->
     </template>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User input → labelEdits | User types/selects field values in inline editors; must validate before commit |
| labelEdits → mutableEntries | When commitLabelEdits runs, apply validated edits to in-memory state |
| mutableEntries → File export | Later (Plan 02) when exporting JSON, file contains user-edited data |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Input Validation | BenchmarkTable inline inputs | mitigate | Validate field values in useLabeling.validateEditField before storing in labelEdits; show validation errors inline; disable commit if hasValidationErrors is true |
| T-06-02 | Data Integrity | labelEdits state mutation | mitigate | Use immutable update patterns (spread operators, not direct mutations); validate before commitLabelEdits applies to mutableEntries |
| T-06-03 | User Confusion | Labeling mode state | mitigate | Button text changes ("✏ Label" ↔ "✓ Done") to indicate mode; inline error messages guide user to fix validation issues |
</threat_model>
