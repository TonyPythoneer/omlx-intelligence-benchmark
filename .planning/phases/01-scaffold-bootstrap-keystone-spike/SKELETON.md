## Architecture & Tech Stack

### Core Stack

| Component | Choice | Rationale |
|-----------|--------|-----------|
| **Framework** | Vue 3 + Composition API | Simple, type-safe, lightweight for one-page app |
| **Build Tool** | Vite+ (`@voidzero-dev/vite-plus-core`) | Already wired; `vp dev` / `vp test` / `vp build` commands |
| **Output** | Static SPA (plain Vue, **not vite-ssg**) | One page; no router/layouts/prerender needed; avoid build-time browser API execution |
| **SSR/Prerender** | None; client-rendered SPA | Hostname guard / File System Access / clipboard all run in browser |
| **Language** | TypeScript strict mode | Type-safe schema (types/benchmark.ts); prevents late refactors |
| **Styling** | Scoped SFC `<style scoped>` | 1:1 CSS port from vanilla app; no Tailwind in v1 |
| **State Management** | Composition API composables | No Pinia; lightweight for single-page scope; e.g. `useTableState()`, `useFilters()`, `useImportState()` |
| **Testing** | Vitest (existing `vp test`) | Ported parser tests; Phase 3+ adds component tests |
| **CI** | Per-phase Playwright selectors | Validate UI changes phase-by-phase, not all at Phase 7 |
| **Deployment** | Static file hosting (serverless) | No runtime server; `cd-static.yml` workflow unchanged |

### Data Contract

**Format:** Pure JSON arrays in `app/data/*.json` and `app/settings.json`

**Entry schema** (from types/benchmark.ts):
```typescript
interface Entry {
  model: string
  date: string (YYYY-MM-DD)
  spec: {
    parameters_b: number
    quantization: string
    size_gb: number
  }
  deprecated: boolean
  tiers: {
    opus: boolean
    sonnet: boolean
    haiku: boolean
  }
  scores: Record<string, {
    accuracy: number
    samples: number
    time_s: number
  }>
  labelling?: { tiers }  // future
  abilities?: { thinking, mtp: boolean }  // future
}
```

**Immutable constraints:**
- No schema changes to existing `app/data/*.json` files
- Parser logic (`parseImportInput`, `mergeImport`) ported as-is
- "Overwrite scores only on duplicate model" rule preserved
- File System Access save model (user-confirmed, no server persistence)

### Key Layout Decisions

**Branch isolation:** All work on `feat/vue-vite-static-site`; main branch remains untouched with vanilla app. Phase 7 performs an atomic swap (replace app/index.html with dist/index.html output).

**Vite root: 'src'** separates the new Vue app from the legacy `app/`. This prevents clobbering the working app during development.

**vp dev** serves localhost:8080 from src/ with hot-reload.  
**vp test** runs vitest on lib/**/*.test.mjs (relative to root: 'src').  
**vp build** produces static dist/ with no server required.

## Key Decisions (Locked for Phases 2–7)

### 1. Plain SPA, Not SSG

**Decision:** Build the app as a client-rendered Vue SPA via `vp build`, not a static-site generator (vite-ssg, unplugin-vue-router, etc.).

**Rationale:**
- One page, no complex routing → SPA is simpler
- Hostname guard (`window.location.hostname`) is read at runtime, not build time (critical for import button visibility)
- File System Access API (`showSaveFilePicker`) runs client-only; no prerender/build-time execution
- Still produces fully static, serverless output via vite's standard build
- Avoids the complexity of SSG plugins for a single-page app

**Implication:** `vp build` → `dist/` is plain HTML/JS/CSS, no special SSG integration needed. If this fails, the issue is fundamental (not a Phase 2 problem).

### 2. Composition API (No Pinia)

**Decision:** Manage state via `useXxx()` composables (Composition API), not Pinia.

**Rationale:**
- Single-page scope: no complex, deeply nested store
- Lightweight, type-safe composition → easier to test
- Less boilerplate than Pinia for simple cases
- Flexibility: composables are just functions; no centralized store constraints

**Patterns:**
- `useTableState()` → table data + sort state
- `useFilters()` → filter state + logic
- `useImportState()` → staging + validation state
- `useClientOnly()` → browser-only API guard (File System Access, clipboard, hostname)

### 3. TypeScript Types First

**Decision:** Define the Entry schema in `types/benchmark.ts` in Phase 1, before building components.

**Rationale:**
- Single source of truth for data shape
- Components, composables, tests all reference this one type
- Prevents late refactors when data shape changes

**Scope:** Entry, Spec, Tiers, Abilities, ScoreLeaf, Scores. Interfaces are read-only data shapes; no defaults or validators (those come in Phase 3+ when filters/labeling/validation are needed).

### 4. Scoped SFC Styles, 1:1 CSS Port

**Decision:** Style components using scoped `<style scoped>` in .vue files, porting CSS 1:1 from the vanilla app.

**Rationale:**
- Faithful parity migration → preserves visual design
- Scoped styles prevent naming conflicts
- Tailwind deferred to v2 (design enhancement, not migration)
- No external CSS framework overhead in Phase 1

**Implications:**
- Each component has its own `<style>` block
- Global styles (fonts, resets) imported in App.vue or index.html `<style>`
- Color classes and breakpoints copied as-is from vanilla CSS

### 5. Browser-Only APIs via `useClientOnly()`

**Decision:** Establish a `useClientOnly()` composable that guards File System Access, clipboard, and hostname checks.

**Pattern:**
```typescript
const { isBrowser } = useClientOnly()
if (isBrowser) {
  // File System Access / clipboard / location.hostname
}
```

**Rationale:**
- Moot under SPA (no prerender), but establishes pattern if SSG is reconsidered
- Defensive: keeps code path clear for browser-only APIs
- Used in Phase 5 (import button) and Phase 6 (export/save)

### 6. Per-Phase Playwright Validation

**Decision:** Validate UI changes incrementally (Phases 3–6 each validate their own Playwright checkpoints), not all at Phase 7.

**Rationale:**
- Phase 3: table rendering checkpoints (selectors for three-tier header, sort)
- Phase 4: filter checkpoints (search, tier, metrics, params, deprecated)
- Phase 5: import modal checkpoints
- Phase 6: labeling + export checkpoints
- Phase 7: full 9-point regression suite

**Implication:** Playwright selectors are updated as the DOM changes; Phase 7 is validation-only, not discovery.

### 7. Atomic Swap at Phase 7

**Decision:** Keep the legacy `app/index.html` untouched until Phase 7, then replace it with the built Vue SPA in a single atomic commit.

**Rationale:**
- Branch isolation: `feat/vue-vite-static-site` never breaks the main app
- Phase 7 is low-risk: just file replacement + CI validation
- Deployment (`cd-static.yml`) remains unchanged; static output replaces the vanilla app

**Swap Mechanism:**
```bash
# Phase 7 final commit:
rm app/index.html app/lib/  # Remove vanilla app
cp dist/index.html app/index.html
cp dist/*.js dist/*.css app/  # or adjust paths based on dist structure
vite.config.ts root: 'app'   # Switch back to app/ as root
```

## Assumptions & Risks

### Locked Assumptions (prove in Phase 1)
1. `vp build` produces a static SPA bundle that runs without a server
2. Vue SFC rendering works end-to-end in the Vite+ pipeline
3. Ported parser tests run green under vitest

### Open Risks (Phase 1 mitigates)
1. **vp build complexity** — If vite.config.ts requires special handling for static output, resolve before Phase 2

### Deferred (v2 / future phases)
- vite-ssg, unplugin-vue-router, vite-plugin-vue-layouts
- Tailwind CSS (scoped CSS sufficient for parity)
- Component library / design system
- Advanced features (theme switching, multi-device comparison UX, etc.)
